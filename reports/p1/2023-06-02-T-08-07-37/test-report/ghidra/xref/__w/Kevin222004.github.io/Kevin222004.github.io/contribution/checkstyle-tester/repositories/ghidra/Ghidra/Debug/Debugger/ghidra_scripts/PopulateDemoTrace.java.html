46  /**
47   * This script populates a trace database for demonstrations purposes and opens it in the current
48   * tool.
49   * 
50   * <p>
51   * Your current tool had better be the "TraceBrowser"! The demonstration serves two purposes. 1) It
52   * puts interesting data into the TraceBrowser and leaves some annotations as an exercise. 2) It
53   * demonstrates how a decent portion the Trace API works.
54   * 
55   * <p>
56   * A Trace is basically a collection of observations of memory and registers over the lifetime of an
57   * application or computer system. In Ghidra, the Trace object also supports many of the same
58   * annotations as does Program. In the same way that Program brings knowledge markup to an image of
59   * bytes, Trace brings knowledge markup to bytes observed over time.
60   * 
61   * <p>
62   * Effectively, if you take the cross-product of Program with time and add Threads, Breakpoints,
63   * etc., you get Trace. It's a lot. In order to use all the UI components which take a Program,
64   * Trace can present itself as a Program at a particular point in time.
65   * 
66   * <p>
67   * Each particular component will be introduced as its used in the script below, but for now some
68   * core concepts:
69   * 
70   * <ul>
71   * <li>A point in time is called a "snap." These don't necessarily correspond to any real unit of
72   * time, though they may. The only requirement is that they are numbered in chronological
73   * order.</li>
74   * <li>Every annotation has a "lifespan," which is the range of snaps for which the annotation is
75   * effective. Some annotations may overlap, others may not. In general, if the corresponding concept
76   * in Program permits address overlap, then Trace permits both address and time overlap. If not,
77   * then neither is permitted. In essense, Trace defines overlap as the intersection of rectangles,
78   * where an annotation's X dimension is it's address range, and its Y dimension is its lifespan.
79   * </li>
80   * <li>Observations in memory happen at a particular snap and are assumed in effect until another
81   * observation changes that. To record the "freshness" of observations, the memory manager tags
82   * regions as KNOWN, UNKNOWN, or ERROR. An observation implicitly marks the affected region as
83   * KNOWN. The intent is to grey the background for regions where memory is UNKNOWN for the current
84   * snap.</li>
85   * <li>Observations of registers behave exactly the same as observations for memory, by leveraging
86   * Ghidra's "register space." The only difference is that those observations must be recorded with
87   * respect to a given thread. Each thread is effectively allocated its own copy of the register
88   * space. Most the the API components require you to obtain a special "register space" for a given
89   * thread before recording observations of or applying annotations to that thread.</li>
90   * </ul>
91   * 
92   * <p>
93   * After you've run this script, a trace should appear in the UI. Note that there is not yet a way
94   * to save a trace in the UI. As an exercise, try adding data units to analyze the threads' stacks.
95   * It may take some getting accustomed to, but the rules for laying down units should be very
96   * similar to those in a Program. However, the Trace must take the applied units and decide how far
97   * into the future they are effective. In general, it defaults to "from here on out." However, two
98   * conditions may cause the trace to choose an ending tick: 1) The underlying bytes change sometime
99   * in the future, and 2) There is an overlapping code unit sometime in the future.
100  * 
101  * <p>
102  * The trace chooses the latest tick possible preceding any byte change or existing code unit, so
103  * that the unit's underlying bytes remain constant for its lifespan, and the unit does not overlap
104  * any existing unit. This rule causes some odd behavior for null-terminated strings. I intend to
105  * adjust this rule slightly for static data types wrt/ byte changes. For those, the placed unit
106  * should be truncated as described above, however, another data unit of the same type can be placed
107  * at the change. The same rule is then applied iteratively into the future until an overlapping
108  * unit is encountered, or there are no remaining byte changes.
109  */

190 	/**
191 	 * Set RIP at the given tick for the given space to the address of a given instruction
192 	 * 
193 	 * @param tick the tick
194 	 * @param regs the register space for a given thread
195 	 * @param ins the instructions
196 	 */
197 	protected void putRIP(long tick, TraceMemorySpace regs, Instruction ins) {
198 		regs.setValue(tick,
199 			new RegisterValue(reg("RIP"), ins.getAddress().getOffsetAsBigInteger()));
200 	}

202 	/**
203 	 * (Re-)place a data unit if necessary.
204 	 * 
205 	 * This is a TODO item in the Trace database. Currently, the API allows a caller to modify bytes
206 	 * in the middle of a code unit's lifespan. The intended rule is: If the modification is at the
207 	 * unit's start tick, the behavior should be the same as Program (permit changes under static
208 	 * data types only). If the modification is after, then the code unit's lifespan should be
209 	 * truncated to allow the modification. Additionally, for static data types, a new unit should
210 	 * be placed to fill out the old lifespan.
211 	 * 
212 	 * Because that TODO is not implemented yet, this method corrects the issue by implementing
213 	 * effectively the same rule.
214 	 * 
215 	 * @param tick the tick where a register change may have occurred
216 	 * @param thread the thread whose registers to check/correct
217 	 * @param reg the register to check/correct
218 	 * @throws CodeUnitInsertionException shouldn't happen
219 	 * @throws CancelledException shouldn't happen
220 	 */
221 	protected void placeRegUnitIfNeeded(long tick, TraceThread thread, Register reg)
222 			throws CodeUnitInsertionException, CancelledException {
223 		// NOTE: This is compensating for a TODO in the memory and code managers
224 
225 		// TODO: Consider convenience methods TraceThread#getMemorySpace(boolean), etc
226 		TraceMemorySpace mem =
227 			thread.getTrace().getMemoryManager().getMemoryRegisterSpace(thread, true);
228 		// First check if the value was set at all
229 		if (mem.getState(tick, reg) != TraceMemoryState.KNOWN) {
230 			return;
231 		}
232 		// The value may have been set, but not changed
233 		RegisterValue oldValue = mem.getValue(tick - 1, reg);
234 		RegisterValue newValue = mem.getValue(tick, reg);
235 		if (Objects.equals(oldValue, newValue)) {
236 			return;
237 		}
238 
239 		TraceCodeSpace code =
240 			thread.getTrace().getCodeManager().getCodeRegisterSpace(thread, true);
241 		code.definedUnits().clear(Lifespan.nowOn(tick), reg, TaskMonitor.DUMMY);
242 		code.definedData().create(Lifespan.nowOn(tick), reg, PointerDataType.dataType);
243 	}
244 
245 	/**
246 	 * Invoke the above method for the three registers I modify during this demonstration.
247 	 * 
248 	 * @param tick the tick where the registers may have changed
249 	 * @param thread the thread to check/correct
250 	 * @throws CodeUnitInsertionException shouldn't happen
251 	 * @throws CancelledException shoudn't happen
252 	 */
253 	protected void placeRegUnits(long tick, TraceThread thread)
254 			throws CodeUnitInsertionException, CancelledException {
255 		placeRegUnitIfNeeded(tick, thread, reg("RIP"));
256 		placeRegUnitIfNeeded(tick, thread, reg("RSP"));
257 		placeRegUnitIfNeeded(tick, thread, reg("RBP"));
258 	}

260 	@Override
261 	protected void run() throws Exception {
262 		/**
263 		 * Construct a Trace with x86_64 and the default compiler
264 		 */
265 		langServ = DefaultLanguageService.getLanguageService();
266 		x86Lang = langServ.getLanguage(new LanguageID("x86:LE:64:default"));
267 		defaultSpace = x86Lang.getAddressFactory().getDefaultAddressSpace();
268 		cspec = x86Lang.getDefaultCompilerSpec();
269 		trace = new DBTrace("Demo", cspec, this);
270 
271 		/**
272 		 * Grab the memory manager and global namespace symbol for convenience
273 		 */
274 		memory = trace.getMemoryManager();
275 		global = trace.getSymbolManager().getGlobalNamespace();
276 
277 		/**
278 		 * A list of instructions in the main function for ease of setting RIP
279 		 */
280 		List<Instruction> mainInstructions = new ArrayList<>();
281 
282 		/**
283 		 * Instead of tracking this by hand, use variables!
284 		 */
285 		int stack1offset = 0;
286 		int stack2offset = 0;
287 		int pc1 = 0;
288 		int pc2 = 0;
289 
290 		/**
291 		 * For clarity, I will add each tick to the trace in its own transaction. The
292 		 * Transaction class eases the syntax and reduces errors in starting and ending
293 		 * transactions.
294 		 */
295 		try (Transaction tx = trace.openTransaction("Populate First Snapshot")) {
296 			/**
297 			 * While not strictly required, each tick should be explicitly added to the database and
298 			 * given a description. Some things may mis-behave if there does not exist at least one
299 			 * tick.
300 			 */
301 			TraceSnapshot snapshot = trace.getTimeManager().createSnapshot("Launched");
302 			long snap = snapshot.getKey();
303 
304 			/**
305 			 * Add two regions to the trace: ".text" for the program instructions and "[STACK 1]"
306 			 * for the main thread's stack. Again, while not strictly required all memory
307 			 * observations should occur within a recorded region. As regions may come and go
308 			 * through the course of execution, this is the first place where we see a lifespan.
309 			 * These are represented using a Range object from Guava. While you may specify open or
310 			 * closed ends, the range will be normalized to use closed ends. Also Long.MIN and
311 			 * Long.MAX will be normalized to negative and positive infinity, respectively. In
312 			 * general, observations should be given the range [currentTick..INF) meaning "from here
313 			 * on out". Most annotations allow mutation of the end tick.
314 			 * 
315 			 * The trace database DOES permit recording and retrieving observations outside any
316 			 * recorded region. However, when viewed as a Program, only the current regions are
317 			 * presented as memory blocks. Thus, observations outside a region are not visible in
318 			 * the UI.
319 			 */
320 			memory.addRegion(".text", Lifespan.nowOn(snap), rng(0x00400000, 0x00400fff),
321 				TraceMemoryFlag.READ, TraceMemoryFlag.EXECUTE);
322 			memory.addRegion("[STACK 1]", Lifespan.nowOn(snap), rng(0x00100000, 0x001effff),
323 				TraceMemoryFlag.READ, TraceMemoryFlag.WRITE);
324 
325 			/**
326 			 * Create the main thread, assumed alive from here on out.
327 			 */
328 			thread1 = trace.getThreadManager().addThread("Thread 1", Lifespan.nowOn(snap));
329 			/**
330 			 * Get a handle to the main thread's register values.
331 			 * 
332 			 * Note that the values are accessed via the memory manager. The memory implementation
333 			 * is re-used to record register values, since registers to Ghidra are just special
334 			 * addresses. As a convenience, the register-space interface of the memory manager
335 			 * presents conveniences which use Register and RegisterValue instead of Address and
336 			 * ByteBuffer.
337 			 */
338 			regs1 = memory.getMemoryRegisterSpace(thread1, true);
339 
340 			/**
341 			 * Place these labels. I guessed and checked as far as positions, go. There just needs
342 			 * to be enough room in main for the assembled instructions below. "child" has to be
343 			 * placed more carefully. Perhaps in a later version of this demo, I will upgrade main,
344 			 * clone, and exit to actual functions.
345 			 */
346 			mainLabel = trace.getSymbolManager()
347 					.labels()
348 					.create(snap, null, addr(0x00400000),
349 						"main", global, SourceType.USER_DEFINED);
350 			cloneLabel = trace.getSymbolManager()
351 					.labels()
352 					.create(snap, null, addr(0x00400060),
353 						"clone", global, SourceType.USER_DEFINED);
354 			childLabel = trace.getSymbolManager()
355 					.labels()
356 					.create(snap, null, addr(0x00400034),
357 						"child", global, SourceType.USER_DEFINED);
358 			exitLabel = trace.getSymbolManager()
359 					.labels()
360 					.create(snap, null, addr(0x00400061),
361 						"exit", global, SourceType.USER_DEFINED);
362 
363 			/**
364 			 * Note the use of getProgramView as a means of using components intended for Program
365 			 * with Trace. Such views typically have a fixed tick, however, you can also obtain a
366 			 * view which can seek to a different tick. The variable-tick version is generally for
367 			 * UI compatibility, whereas the fixed-tick version is generally for API compatibility.
368 			 * Here we use it to apply the assembler.
369 			 * 
370 			 * This is the "main" function of the demonstration. it is imagined with a decent bit of
371 			 * fidelity, but some bits are elided for my sanity and to stay succinct.
372 			 * 
373 			 * It starts with the typical stack frame setup and then immediately clones a second
374 			 * thread. I arbitrarily decided which thread would execute for each step. The cloned
375 			 * thread then jumps to the "child" portion of the code while the main thread falls
376 			 * through. Each places some data on its stack and then terminate. The "clone" and
377 			 * "exit" functions are stubbed out, but one should imagine they are system calls.
378 			 * 
379 			 * A call to "clone" results in the creation of a second thread with stack. That stack
380 			 * contains only the return address. The caller's RAX is set to 0, the clone's RAX is
381 			 * set to 1.
382 			 * 
383 			 * A call to "exit" results in the immediate termination of the calling thread.
384 			 */
385 			Assembler asm = Assemblers.getAssembler(trace.getFixedProgramView(snap));
386 			Iterator<Instruction> mainBlock = asm.assemble(mainLabel.getAddress(), //
387 				"PUSH RBP", //
388 				"MOV RBP,RSP", //
389 				"CALL clone", //
390 				"TEST EAX,EAX", //
391 				"JNZ child", //
392 				"SUB RSP,0x10", //
393 				"MOV dword ptr [RSP],0x6c6c6548", //
394 				"MOV dword ptr [RSP+4],0x57202c6f", //
395 				"MOV dword ptr [RSP+8],0x646c726f", //
396 				"MOV word ptr [RSP+0xc],0x21", //
397 				"CALL exit", //
398 				"SUB RSP,0x10", //
399 				"MOV dword ptr [RSP],0x2c657942", //
400 				"MOV dword ptr [RSP+4],0x726f5720", //
401 				"MOV dword ptr [RSP+8],0x21646c", //
402 				"CALL exit" //
403 			);
404 			mainBlock.forEachRemaining(mainInstructions::add);
405 
406 			/**
407 			 * Stub out "clone"
408 			 */
409 			asm.assemble(cloneLabel.getAddress(), "RET");
410 			trace.getCodeManager()
411 					.codeUnits()
412 					.getAt(0, cloneLabel.getAddress())
413 					.setComment(
414 						CodeUnit.EOL_COMMENT, "Pretend this is a syscall");
415 
416 			/**
417 			 * Stub out "exit"
418 			 */
419 			asm.assemble(exitLabel.getAddress(), "HLT");
420 			trace.getCodeManager()
421 					.codeUnits()
422 					.getAt(0, exitLabel.getAddress())
423 					.setComment(
424 						CodeUnit.EOL_COMMENT, "Pretend this is a syscall");
425 
426 			/**
427 			 * "Launch" the program by initializing RIP and RSP of the main thread
428 			 */
429 			putRIP(snap, regs1, mainInstructions.get(pc1));
430 			regs1.setValue(snap,
431 				new RegisterValue(reg("RSP"), BigInteger.valueOf(STACK1_BOTTOM + stack1offset)));
432 
433 			placeRegUnits(snap, thread1);
434 		}
435 
436 		/**
437 		 * Just hand emulate the stepping
438 		 */
439 		try (Transaction tx = trace.openTransaction("Step")) {
440 			long snap = trace.getTimeManager().createSnapshot("Stepped: PUSH RBP").getKey();
441 
442 			stack1offset -= 8;
443 			putRIP(snap, regs1, mainInstructions.get(++pc1));
444 			/**
445 			 * This demonstrates recording a memory observation, i.e., writing to trace memory.
446 			 * 
447 			 * The ByteBuffer API can be a bit annoying, but it generally follows the same pattern,
448 			 * and it permits packing an arbitrary number of "fields" into the buffer (limited by
449 			 * the allocated size of the buffer). If this API is too inconvenient, you can also use
450 			 * getBufferAt, to use the familiar MemBuffer API.
451 			 */
452 			memory.putBytes(snap, addr(STACK1_BOTTOM - 8), buf.clear().putLong(0).flip());
453 			/**
454 			 * Since register "memory" is just an extension of physical memory, the same API is
455 			 * available for recording register observations.
456 			 */
457 			regs1.putBytes(snap, reg("RSP"),
458 				buf.clear().putLong(STACK1_BOTTOM + stack1offset).flip());
459 
460 			placeRegUnits(snap, thread1);
461 		}
462 
463 		/**
464 		 * More hand emulation
465 		 */
466 		try (Transaction tx = trace.openTransaction("Step")) {
467 			long snap = trace.getTimeManager().createSnapshot("Stepped: MOV RBP,RSP").getKey();
468 
469 			putRIP(snap, regs1, mainInstructions.get(++pc1));
470 			regs1.putBytes(snap, reg("RBP"),
471 				buf.clear().putLong(STACK1_BOTTOM + stack1offset).flip());
472 
473 			placeRegUnits(snap, thread1);
474 		}
475 
476 		/**
477 		 * Emulate the clone "syscall"
478 		 * 
479 		 * While this is a complicated call, there is nothing new to demonstrate in its
480 		 * implementation. As an exercise, see if you can follow what is happening within.
481 		 */
482 		try (Transaction tx = trace.openTransaction("Step")) {
483 			long snap = trace.getTimeManager()
484 					.createSnapshot("Stepped Thread 1: CALL clone -> Thread 2")
485 					.getKey();
486 
487 			memory.addRegion("[STACK 2]", Lifespan.nowOn(snap), rng(0x00200000, 0x002effff),
488 				TraceMemoryFlag.READ, TraceMemoryFlag.WRITE);
489 
490 			thread2 = trace.getThreadManager().addThread("Thread 2", Lifespan.nowOn(snap));
491 			regs2 = memory.getMemoryRegisterSpace(thread2, true);
492 
493 			stack1offset -= 8;
494 			regs1.putBytes(snap, reg("RIP"),
495 				buf.clear().putLong(cloneLabel.getAddress().getOffset()).flip());
496 			regs1.putBytes(snap, reg("RSP"),
497 				buf.clear().putLong(STACK1_BOTTOM + stack1offset).flip());
498 			regs1.putBytes(snap, reg("RAX"), buf.clear().putLong(0).flip());
499 			memory.putBytes(snap, addr(STACK1_BOTTOM + stack1offset),
500 				buf.clear().putLong(mainInstructions.get(++pc1).getAddress().getOffset()).flip());
501 
502 			stack2offset -= 8;
503 			regs2.putBytes(snap, reg("RIP"),
504 				buf.clear().putLong(cloneLabel.getAddress().getOffset()).flip());
505 			regs2.putBytes(snap, reg("RSP"),
506 				buf.clear().putLong(STACK2_BOTTOM + stack2offset).flip());
507 			regs2.putBytes(snap, reg("RAX"), buf.clear().putLong(1).flip());
508 			memory.putBytes(snap, addr(STACK2_BOTTOM + stack2offset), buf.clear()
509 					.putLong(
510 						mainInstructions.get(pc2 = pc1).getAddress().getOffset())
511 					.flip());
512 
513 			placeRegUnits(snap, thread1);
514 			placeRegUnits(snap, thread2);
515 		}
516 
517 		/**
518 		 * Hand emulate thread1 a few steps
519 		 */
520 		try (Transaction tx = trace.openTransaction("Step")) {
521 			long snap =
522 				trace.getTimeManager().createSnapshot("Stepped Thread 1: RET from clone").getKey();
523 
524 			stack1offset += 8;
525 			putRIP(snap, regs1, mainInstructions.get(pc1));
526 			regs1.putBytes(snap, reg("RSP"),
527 				buf.clear().putLong(STACK1_BOTTOM + stack1offset).flip());
528 
529 			placeRegUnits(snap, thread1);
530 		}
531 
532 		/**
533 		 * ...
534 		 */
535 		try (Transaction tx = trace.openTransaction("Step")) {
536 			long snap =
537 				trace.getTimeManager().createSnapshot("Stepped Thread 1: TEST EAX,EAX").getKey();
538 
539 			putRIP(snap, regs1, mainInstructions.get(++pc1));
540 			regs1.putBytes(snap, reg("ZF"), buf.clear().put((byte) 1).flip());
541 
542 			placeRegUnits(snap, thread1);
543 		}
544 
545 		try (Transaction tx = trace.openTransaction("Step")) {
546 			long snap =
547 				trace.getTimeManager().createSnapshot("Stepped Thread 1: JNZ child").getKey();
548 
549 			putRIP(snap, regs1, mainInstructions.get(++pc1));
550 
551 			placeRegUnits(snap, thread1);
552 		}
553 
554 		/**
555 		 * Switch to thread2
556 		 */
557 		try (Transaction tx = trace.openTransaction("Step")) {
558 			long snap =
559 				trace.getTimeManager().createSnapshot("Stepped Thread 2: RET from clone").getKey();
560 
561 			stack2offset += 8;
